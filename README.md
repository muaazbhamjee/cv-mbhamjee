# Academic Documents — Prof. Muaaz Bhamjee

LaTeX document suite with a Python-driven publication system. Research outputs
are stored in BibTeX files and automatically injected into the CV at build time.
All documents are compiled by a single build script and published via GitHub Pages.

---

## Quick start

```bash
# 1. Clone
git clone https://github.com/<your-username>/cv-bhamjee.git
cd cv-bhamjee

# 2. Install dependencies (once — see Dependencies below)

# 3. Build everything
make

# 4. Open PDFs
make open
```

```
│   ├── cv_bhamjee.pdf        #   Stable public URL (see GitHub Pages below)

```

> **Root-level PDFs** are local working copies — rebuilt on demand, never committed.
> **`docs/` PDFs** are deliberate publish snapshots — committed and served via GitHub Pages.

## Make targets

| Command | Action |
|---|---|
| `make` | Build all documents |
| `make cv` | Build CV only |
| `make statement` | Build NITheCS Candidate Statement only |
| `make clean` | Remove LaTeX artefacts — **PDFs are preserved** |
| `make clean-all` | Remove artefacts and all generated PDFs |
| `make open` | Build all and open both PDFs |
| `make open-cv` | Build and open CV only |
| `make open-stmt` | Build and open NITheCS statement only |
| `make publish msg="..."` | Build, stage `docs/`, commit, and push |
| `make watch` | Rebuild automatically on any source change |

## How to update publications

### 1. Find the right `.bib` file

| Type | File |
|---|---|
| Direct journal paper | `journals_non_atlas.bib` |
| ATLAS / CERN paper | `journals_atlas.bib` |
| Conference paper | `conferences.bib` |
| Patent | `patents.bib` |
| Book chapter | `bookchapters.bib` |
| Talk, poster, software, dataset | `other_scholarly.bib` |

### 2. Paste in the BibTeX entry

Most journals and Google Scholar provide a "Cite → BibTeX" export button.
Order within the file doesn't matter — `build.py` sorts by year automatically.

**Journal paper:**
```bibtex
@article{bhamjee2026newpaper,
  author  = {Bhamjee, Muaaz and Connell, Simon H.},
  title   = {A New Result on Multiphase Flow},
  journal = {Journal of Fluid Mechanics},
  volume  = {999},
  pages   = {1--20},
  year    = {2026},
  doi     = {10.xxxx/xxxxxx},
}
```

**Conference paper:**
```bibtex
@inproceedings{bhamjee2026conf,
  author    = {Bhamjee, Muaaz and Smith, Jane},
  title     = {Deep Learning for CFD Surrogate Models},
  booktitle = {SACAM 2026},
  address   = {Cape Town},
  pages     = {1--8},
  year      = {2026},
}
```

### 3. Rebuild and commit

```bash
make cv
git add journals_non_atlas.bib   # or whichever file changed
git commit -m "Add JFM paper 2026"
git push
```

## How to update ATLAS papers

ATLAS output is prolific. Recommended cadence: **once or twice a year**.

1. Go to your [Google Scholar profile](https://scholar.google.co.za/citations?user=jlOJkoAAAAJ)
2. Select all → Export → **BibTeX**
3. Copy new ATLAS entries into `journals_atlas.bib`
4. Run `make cv` — duplicates won't appear (each entry has a unique key)

## How to add a new document

1. Add a `.tex` file to the repository root.
2. Add one entry to the `DOCUMENTS` registry in `build.py`:

```python
{
    'name':  'nrf_motivation',
    'type':  'latex',
    'tex':   os.path.join(SCRIPT_DIR, 'nrf_motivation.tex'),
    'label': 'NRF Rating Motivation',
},
```

3. Add a corresponding entry to the Makefile's `STMT_SOURCES` or define its own variable.
4. Run `make` — the new document is built, and a copy lands in `docs/` automatically.

## Publishing to GitHub Pages

Every successful `make` build copies all PDFs into `docs/`. That folder is tracked
by Git, so you control exactly when a new version goes public.

**To publish:**
```bash
# One-liner with make:
make publish msg="Update CV and NITheCS statement March 2026"

# Or manually:
make
git add docs/
git commit -m "Publish March 2026"
git push
```

**GitHub Pages setup (one-time):**
1. Go to your repo on GitHub → **Settings → Pages**
2. Source: **Deploy from a branch**
3. Branch: `main`, Folder: `/docs`
4. Save

Your documents will then be available at stable public URLs:
```
https://<your-username>.github.io/cv-bhamjee/cv_bhamjee.pdf
https://<your-username>.github.io/cv-bhamjee/nithecs_statement.pdf
```

These URLs are safe to share in email signatures, Google Scholar, LinkedIn, etc.

## Windows + WSL setup

If you are running VS Code via WSL but have LaTeX installed on Windows
(TeX Live or MiKTeX), point `build.py` at the Windows binary rather than
installing LaTeX a second time inside WSL.

### Step 1 — Find your Windows pdflatex path

Open **PowerShell** on Windows and run:
```powershell
where.exe pdflatex
# e.g. C:\texlive\2024\bin\windows\pdflatex.exe
#   or C:\Users\Muaaz\AppData\Local\Programs\MiKTeX\miktex\bin\x64\pdflatex.exe
```

### Step 2 — Add it to your WSL shell profile

Convert the Windows path to its WSL equivalent (`C:\` → `/mnt/c/`) and add
this line to `~/.bashrc` (or `~/.zshrc`):

```bash
# TeX Live
export PDFLATEX=/mnt/c/texlive/2024/bin/windows/pdflatex.exe

# MiKTeX (replace <YourUsername> with your Windows username)
export PDFLATEX="/mnt/c/Users/<YourUsername>/AppData/Local/Programs/MiKTeX/miktex/bin/x64/pdflatex.exe"
```

Then reload your shell:
```bash
source ~/.bashrc
```

### Step 3 — Build as normal

```bash
make       # build.py picks up $PDFLATEX automatically
```

`build.py` also auto-detects TeX Live and MiKTeX at their default locations
without needing the environment variable — the `export` is only necessary if
your install is in a non-standard location.

### Override without editing .bashrc

You can also pass the path inline for a one-off build:
```bash
make PDFLATEX=/mnt/c/texlive/2024/bin/windows/pdflatex.exe
```

### Opening PDFs from WSL

`make open` will try `wslview` first (from `wslu`, the recommended WSL
utilities package), then fall back to `explorer.exe`. Install `wslu` for the
best experience:
```bash
sudo apt install wslu
```

Then `make open` opens the PDF in your default Windows PDF viewer.

## Dependencies

| Tool | Purpose | Install |
|---|---|---|
| Python 3.6+ | Runs `build.py` | Usually pre-installed |
| `pdflatex` | Compiles LaTeX → PDF | See below |
| `fswatch` *(optional)* | `make watch` on macOS | `brew install fswatch` |
| `inotify-tools` *(optional)* | `make watch` on Linux | `apt install inotify-tools` |

**LaTeX installation:**

```bash
# macOS
brew install --cask mactex-no-gui   # ~4 GB full install
brew install --cask basictex        # ~100 MB minimal

# Ubuntu / Debian
sudo apt install texlive-latex-recommended texlive-fonts-recommended \
                 texlive-latex-extra

# Arch
sudo pacman -S texlive-most
```

## VS Code setup

Install **LaTeX Workshop** (`james-yu.latex-workshop`). Add the following to
`settings.json` to route builds through `make` rather than its own chain:

```json
"latex-workshop.latex.recipes": [
  { "name": "make", "tools": ["make"] }
],
"latex-workshop.latex.tools": [
  { "name": "make", "command": "make", "args": [], "env": {} }
]
```

Saving any source file will then trigger a full rebuild automatically.
